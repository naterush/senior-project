Evaluation Metrics

Thanks for checking out Sapling! If you're interested in reading about how our project performs, check out the summary statistics below.

Website Load-times

Using a tool called Pingdom, which is a website speed test to find page load time, analyze it, and find bottlenecks, we found the load time and performance grade of our static web pages. Our landing page had a load time of 158 ms, which is considered excellent, whereas our map page had a load time of 349 ms, which is still within an acceptable range, and comparable to the load time of Google Maps.

Model Runtime Per Step

This metric tests the runtime of just our model at the granularity of each step. We used basic python timing tools to run the model five times in different locations and record the average time that each step took. Within the total average runtime of 21.4 minutes, about 70% of this time is spend within the download step, where the network is the limiting factor. The final change map and JSON output conversion steps take up another 20% of the runtime, but are configurable to run faster at a lower resolution.

Model Accuracy

We evaluated the use of two models on their accuracy. The accuracy of the models is defined here to be the number of correct predictions of tree/no tree normalized by the number of total predictions. We calculated this by testing the model on 5 randomly selected regions in the western and eastern U.S, with the model performing noticeably better in the Western U.S. than the Eastern U.S. This may have been due to biases in our training datasets.

Concurrent Requests Response Time

This metric shows how the full system responds to multiple users requesting new data at once. To test this, we had varying numbers of users simultaneously make un-cached requests to the server, timed how long each request took, and then averaged the response time. The average response time increases by almost 1.5x when there are 4 concurrent requests vs. 1. Furthermore above 4 or more requests to the server simultaneously result in increased error rates and timeouts.

Cache vs. No Cache

This metric represents the speedup of the building of the heatmap that comes from caching the computed growth/death predictions. Without a cache, the runtime is the same as the runtime of the model (21.4 minutes), but upon adding a cache (and once the data is added to the cache), the heatmap can be built in about 12 seconds, which is a speedup of about 100x.
